# Harness Installation — Design Spec

**작성일**: 2026-05-17
**작성자**: brainstorming session (bakpark)
**상태**: Draft (사용자 검토 대기)
**대상 plugin**: `bobs-plugin`

---

## 1. Context

`bobs-plugin` 은 Claude harness 작업 자산 (skill / agent / hook) 의 작성·감사·설계를 묶은 plugin 이다. 현재 다음 자산이 있다.

- 자산 생성기: `skill-creator` ✅, `agent-creator` ✅, `hook-creator` 🚧
- 인덱스·계약 작성기: `agents-md-author`, `context-map-builder` (이 세션 신규), `claude-md-improver` (vendored)
- 설계·감사: `harness-resource-design` (개선 필요), `agent-skill-designer` subagent, `agent-skill-auditor` subagent
- 권위 문서: `references/harness-engineering.md` (640 lines, §1-7)

문제:

1. `harness-engineering.md` 가 개념 모델 (§1-4) 과 배치·체크리스트 (§5-7) 를 한 파일에 섞어 사용 시점·독자가 모호.
2. "하네스 이식" — 빈 또는 기존 프로젝트에 하네스를 도입하는 *워크플로우* 가 자산 사이에 흩어져 있고 entry point 가 불명확.
3. 인덱스·계약 작성기 (`agents-md-author` 등) 가 산출물 단위로 너무 잘게 쪼개져 docs 도메인의 일관성 책임이 분산.
4. design (자원 타입 결정 / 인덱싱 설계 / 검증 인프라 설계) 과 execution (실제 파일 작성) 의 경계가 불명확.

## 2. Goals / Non-goals

**Goals**:
- 하네스 이식 워크플로우를 단일 normative doc 으로 명시
- design ↔ execution 책임 분리 (4 phase / 4 skill 구조)
- 진단 → 동적 시퀀스 (사용자가 빈 / 부분 / 풀 도입 어느 경우든 동일 진입점)
- 메인 세션 skill 체인 모델 (sub-agent 없이)

**Non-goals**:
- Harness-Engineer 라는 별도 sub-agent 도입 (1차 범위 밖 — 향후 옵션)
- 자동 설치 (CLI tool) 화
- 다른 plugin 과의 호환·통합
- backward compatibility (deprecate 되는 기존 자산은 깨끗하게 제거)

## 3. Architecture

### 3.1 두 phase 구조

```
하네스 이식 (Claude 메인 세션 — skill 호출 체인)
│
├── Phase 1 — Diagnose + Design (선택적 호출)
│   각 skill 은 "현황 진단 + 변경 spec" 한 묶음으로 산출
│   ├── resource-design               (skill/agent/hook 타입 결정 + 책임 분리)
│   ├── context-map-architecture      (docs 트리 + AGENTS/CLAUDE/context-map 인덱싱)
│   └── evaluation-loop-design        (roles + evaluation-loop + golden-set + task-log)
│
└── Phase 2 — Execution (자동 dispatch)
    Phase 1 spec 의 Execution Plan 항목별로 dispatch
    ├── skill-creator                 ✅ 기존
    ├── agent-creator                 ✅ 기존
    ├── hook-creator                  🚧 작업중
    └── evaluation-loop-runner        (runtime — task log + gap 라우팅)
```

### 3.2 책임 패턴 비대칭

| 패턴 | Phase 1 skill | Phase 2 skill |
|---|---|---|
| spec → creator 위임 | resource-design | skill-creator / agent-creator / hook-creator |
| spec + 작성 통합 | context-map-architecture | (자체 작성) |
| spec + 작성 통합 | evaluation-loop-design | (자체 작성) |
| runtime 실행 | (없음) | evaluation-loop-runner |

이유:
- 자원 (`skill/agent/hook`) 은 전용 creator + GAP-driven 검증이 깊다 → spec 위임이 자연
- docs 파일은 단순 markdown + 도메인 일관성이 작성 시점에 결정 → design ↔ write 통합이 효율
- 평가 루프 *실행* 은 runtime 동작 (log 캡처, gap 라우팅) — 파일 작성과 분리

### 3.3 데이터 흐름

```
[사용자 요청]
   ↓
[workflow doc routing 표] — 어느 design skill 부터?
   ↓
[Phase 1 design skill] — Inspect → Spec 산출
   ↓
[사용자 spec 검토 + 승인 (design effect gate)]
   ↓
[Spec 의 Execution Plan 파싱]
   ↓
[Phase 2 dispatch] (자동 chain)
   ├── target=skill-creator → skill-creator §2 gate → 파일 작성
   ├── target=agent-creator → ... → 파일 작성
   ├── target=hook-creator → ... → 파일 + settings.json
   └── (context-map-architecture / evaluation-loop-design 자체 작성)
   ↓
[모든 execution 완료]
   ↓
[evaluation-loop-runner — runtime 동안]
   ↓
[gap 발견 → 적절한 design skill 재진입] (사이클)
```

이중 gate:
- Layer 1 (design): spec 산출 시 사용자 승인 → execution dispatch 결정
- Layer 2 (execution): creator skill 내부의 §2 gate (파일 작성 직전) — 추가 안전망

## 4. 신규 4 skill 책임 경계

### 4.1 `resource-design`

**책임**: 새 작업 유형에 어떤 자원 (skill / agent / hook) 이 필요한지 결정 + 기존 자원과의 책임 분리.

**Inspect 도메인**:
- `<repo>/.claude/skills/`, `<repo>/plugins/*/skills/`, `~/.claude/skills/` (skill inventory)
- `<repo>/.claude/agents/`, `<repo>/plugins/*/agents/` (agent inventory)
- `<repo>/.claude/settings.json` (hook 등록)
- 사용자 발화의 작업 컨텍스트

**Spec 산출**:
```markdown
# Harness Installation Spec — resource

> Generated by: resource-design
> Date: <iso8601>
> Trigger: <user request>

## Inventory
## Required Resources       — 작업 유형 → 자원 결정
## Resource Decisions       — 각: name / type / 책임 한 줄 / 우선순위
## Execution Plan           — target / args / rationale
```

**흡수**:
- `harness-resource-design` (개선·재편)
- `agent-skill-designer` subagent (reference 자료)

### 4.2 `context-map-architecture`

**책임**: docs 트리 전반 (`AGENTS.md`, `CLAUDE.md`, `docs/README.md`, `docs/agent/context-map.md`, `docs/architecture.md`, `docs/decisions/`, `docs/domain/`, `docs/integrations/`, `docs/workflows/`, `docs/security.md`) 의 설계 + 작성.

**Inspect 도메인**: 현재 docs 트리 + 인덱싱 누락 + 책임 누수 (e.g., CLAUDE.md 가 AGENTS.md 책임 침범).

**Spec 산출 + 작성**:
```markdown
## Inventory
## Gaps                     — 없는 문서 + 누락 인덱스 + 책임 누수
## Document Plan            — 작성/수정/이동 항목 + 각 파일 골격
## Applied Changes          — 실제 작성/수정한 파일 목록
```

**자체 작성 범위**: 위 docs 전반. 도메인 깊이가 있는 파일 (`docs/architecture.md`, `docs/domain/`) 은 *골격만* 작성하고 본문은 사용자 작업으로 남김. *골격* 의 정의: 섹션 헤더 + 각 섹션 1-2 줄 placeholder + 무엇을 채워야 하는지 안내 코멘트. 본문 prose 자체는 작성하지 않음.

**흡수** (적극적):
- `agents-md-author` — 본문 핵심 (`Phase 1 Inspect / Phase 2 Draft / Phase 3 Effect gate / 8-section template`) 을 `references/agents-md-write.md` 로
- `context-map-builder` — 본문 핵심 (Inventory / Map / Effect gate / template) 을 `references/context-map-write.md` 로
- `claude-md-improver` (vendored, Apache-2.0) — 본문 핵심 절차를 `references/claude-md-write.md` 로 가져옴 + 라이센스 attribution 유지 (THIRD_PARTY_NOTICES.md)

**Effect gate**: 파일 작성 전 Document Plan 표시 + 사용자 승인.

### 4.3 `evaluation-loop-design`

**책임**: `docs/agent/` 검증 인프라 설계 + 작성 (roles.md 본문 / evaluation-loop.md / golden-set.md / task-log-template.md).

**Inspect 도메인**: 현재 `docs/agent/` 의 검증 자산 + 프로젝트 작업 유형 (golden-set 채울 후보) + 기존 task log 패턴.

**Spec 산출 + 작성**:
```markdown
## Inventory
## Gaps
## Verification Plan        — roles 본문 + loop 구조 + golden-set 정의 + log 템플릿
## Applied Changes
```

**자체 작성 범위**: `docs/agent/roles.md` 본문, `docs/agent/evaluation-loop.md`, `docs/agent/golden-set.md`, `docs/agent/task-log-template.md`.

**참고 자료**: `agent-skill-auditor` subagent (검증 도메인 reference).

### 4.4 `evaluation-loop-runner`

**책임**: *runtime* 동작 — 작업 실행 후 task log 캡처 + gap 분석 + 다음 자원 결정 (라우팅).

**입력**: 실행한 작업의 transcript / 명령 / 참조 문서.

**산출**:
```markdown
## Task Log Entry           → docs/agent/logs/YYYY-MM-DD-*.md 추가
## Gap Analysis             — golden-set 비교 결과
## Routing Decision         — docs / agents / skills / hooks 중 어디로 환원
## Next Action              — 호출할 design skill (사이클 진입점)
```

**자동 chain**: `Next Action` 의 design skill 자동 호출 → Phase 1 재진입.

**사이클 종료 조건**: 다음 중 하나일 때 자동 chain 중단:
- gap 분석 결과 `Routing Decision: no-op` (개선할 환원 위치 없음)
- 사용자가 명시 종료 ("stop", "충분", "지금까지")
- 같은 design skill 이 2회 연속 호출됨 (재진입 무한 루프 신호)
- 누적 라운드 5회 초과 (`NEEDS_REVIEW` 로 사용자에게 핸드오프)

## 5. Spec Interface (markdown structured section)

모든 design skill 의 spec 은 공통 헤더 + 4개 표준 섹션:

```markdown
# Harness Installation Spec — <domain>

> Generated by: <skill-name>
> Date: <iso8601>
> Trigger: <user request 또는 runner Routing>

## Inventory          — 현재 상태
## Gaps               — 간격 식별
## Plan               — 변경 제안
## Execution Plan     — forward target skill (Phase 2 dispatch 용)
```

### Execution Plan 파싱 규칙

```markdown
## Execution Plan
- target: skill-creator
  args:
    name: "context-hint-emitter"
    scope: "project"
  rationale: hook trigger 시 짧은 routing hint 주입이 반복 패턴
- target: agent-creator
  args:
    name: "harness-env-maintainer"
  rationale: 환경 개선 specialist 가 별도 컨텍스트 필요
```

Main session 은 이 섹션을 YAML 처럼 파싱해 순차 dispatch. 각 dispatch 는 해당 creator skill 의 §2 (또는 동등) effect gate 를 거친다.

**`args` 형식**: target creator skill 마다 다르다 — 표준화하지 않는다. design skill 은 target 의 frontmatter / 본문을 읽어 args 키 (`name` / `scope` 등) 를 선택. 모르면 비워두고 creator 의 §0 (intent capture) 가 사용자에게 묻도록 한다.

### no-op / blocked case

```markdown
## Execution Plan
mode: no-op
reason: 진단 결과 현재 자원으로 충분
```

```markdown
## Execution Plan
mode: blocked
reason: 자원 결정 모호 — 사용자 입력 필요
needs_input:
  - <질문 1>
  - <질문 2>
```

## 9. Risks

### 9.1 책임 누수 (context-map-architecture 비대)

`context-map-architecture` 가 docs 트리 전반 (10여 개 파일 도메인) 을 다룬다. Skill 본문이 비대해질 위험.

**완화**:
- 본문은 *workflow + 라우팅* 만, 각 파일 유형의 작성 절차는 `references/agents-md-write.md` / `context-map-write.md` / `claude-md-write.md` / `docs-tree-write.md` 로 분리
- 본문 ≤ 300 lines 유지

### 9.2 평가 루프 실행과 hook 의 중첩

`evaluation-loop-runner` 의 task log 캡처가 hook (`task-log-capture.sh`) 과 책임 중첩될 가능성.

**경계**:
- Hook = *자동 트리거* (PostToolUse), 단순 기록
- Runner skill = *명시 호출*, gap 분석 + 라우팅 결정
- 둘은 보완 — hook 이 raw 데이터 수집, runner 가 의미 부여

### 9.3 spec 인터페이스 버전 관리

markdown structured section 이 변하면 모든 design + execution skill 영향. 변경 시 명시적 versioning 필요.

**완화**: spec 헤더에 `spec_version: v1` 필드 명시 + workflow doc §4 에 변경 정책 명시.

### 9.4 dead asset 위험

3개 skill 흡수 후 사용자가 옛 이름으로 호출 시도 가능 (e.g., `/agents-md-author`).

**완화**: 첫 1-2 cycle 동안 deprecated skill 의 SKILL.md 만 stub 으로 유지 ("This skill has been merged into `context-map-architecture`. Use that instead.") + plugin.json 에서 제거.

### 9.5 vendored skill 흡수의 라이센스

`claude-md-improver` 는 Apache-2.0 vendored. 적극 흡수 시 attribution 의무가 새 skill 로 이전.

**완화**: `context-map-architecture/references/claude-md-write.md` 헤더에 "출처: vendored from claude-plugins-official/claude-md-management (Apache-2.0)" 표기 + `THIRD_PARTY_NOTICES.md` 갱신 (skill-creator 가 writing-skills 부분 흡수 시 사용한 패턴 동일).

### 9.6 무한 사이클

`evaluation-loop-runner` → design skill → execution → 다시 runner → design ... 의 자동 chain 이 종료 조건 없으면 무한 진행.

**완화**: §4.4 에 명시한 4가지 종료 조건 (no-op / 사용자 명시 / 동일 design 2회 / 5회 초과). main session 은 사이클 카운터를 유지하고 종료 조건 충족 시 사용자에게 보고.

## 10. Decisions (전 Open Questions, resolved)

1. **`agent-skill-designer` subagent**: **Deprecate**.
   책임이 `resource-design` skill 로 흡수되고, subagent 의 격리 컨텍스트가 *자원 타입 결정* 같은 메인 세션 일상 결정에는 정당화되지 않음. dead asset 위험 회피. → Step 3d 에 자산 삭제 추가.

2. **workflow doc routing 표 (§2) 형식**: **단일 표** 우선.
   신호 → 첫 design skill 의 단순 매핑. 7-10 행 초과해 호출자 판단 비용이 커지면 그때 decision tree (graphviz) 도입 검토.

3. **`claude-md-improver` 흡수 범위**: **핵심 워크플로우만** (Inspect / Draft / Effect gate).
   `context-map-architecture/references/claude-md-write.md` ≤200 lines self-contained 형태. 헤더에 "출처: vendored from claude-plugins-official/claude-md-management (Apache-2.0)" 표기 + `THIRD_PARTY_NOTICES.md` 갱신. skill-creator 가 writing-skills 부분을 references 로 흡수한 패턴 동일.

4. **신규 4 skill 의 GAP 검증 깊이**: **skill-creator 와 동일 GAP-driven 사이클**.
   메인 세션의 자동 dispatch chain 에 들어가는 자산이므로 안전성·신뢰성이 1차 자산 (skill-creator) 과 동일해야 함. 각 Step 의 "GAP 분석 → PASS" 는 `${WORKSPACE}/gaps/*.GAP.md` 산출 + Final Decision PASS / PASS_WITH_NOTES 까지 진행.

5. **`evaluation-loop-runner` 호출 시점**: **명시 호출 우선**.
   - hook (`task-log-capture.sh` 같은 것) = *raw task log 캡처* 만 — runtime 자동, 단순 기록.
   - runner skill = *명시 호출* (사용자 `/runner` 또는 design skill 의 Next Action) — gap 분석 + 라우팅 결정.
   자동 trigger 는 raw data 수집까지, 의미 부여는 명시 호출. §9.2 의 경계와 동일.

6. **commit 전략**: **Step 별 separate commit**.
   각 Step 이 동작 단위 — Step 2 끝나면 context-map-architecture 가 호출 가능 + deprecated 3개 자산 정리 완료. 디버깅·rollback 쉬움. Step 1 (doc split) 은 작아서 Step 2 와 묶어도 가능. version bump 는 모든 Step 완료 후 한 번 (0.2.0 — minor bump, breaking).

## 11. Success Criteria

- 사용자가 "하네스 이식해줘" 라고 했을 때, 메인 세션이 workflow doc 의 routing 을 따라 적절한 design skill 부터 호출
- design skill 이 inspect → spec 산출 → 사용자 승인 → 자동 execution dispatch 의 4 단계로 일관 동작
- 4 신규 skill 의 GAP 가 모두 PASS 또는 PASS_WITH_NOTES
- 기존 deprecate 자산 (3개) 의 핵심 절차가 새 skill 의 references 로 보존
- workflow doc 이 self-contained — 외부 normative source 없이 (harness-principles.md 만으로) 이식 절차 진행 가능
