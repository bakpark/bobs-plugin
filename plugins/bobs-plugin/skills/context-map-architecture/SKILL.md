# Context Map Architecture

프로젝트의 docs 트리를 *설계 + 작성* 하는 절차 메타 스킬. 본 스킬은 *어떤 문서가 어디에 있어야 하는지* 를 진단하고, 작성·갱신해야 할 항목을 Document Plan 으로 묶어 사용자 승인 후 실제 파일 write 까지 처리한다.

`harness-principles` 자산 모델 §4.1 (Docs 책임) + §4.5 (Context Map) + §4.7 (자산 선택 기준) 의 *문서 행* 만 다룬다. skill / agent / hook 같은 *런타임 자원* 결정·작성은 본 스킬 범위 밖.

## When to Use

**Trigger**:

- 사용자가 `AGENTS.md` / `CLAUDE.md` / `docs/agent/context-map.md` / `docs/` 트리 작성·정리·갱신을 요청한다.
- 빈 프로젝트에 "처음부터 하네스" 셋업이 요청된다 (Routing 표 §2 마지막 행 — 인덱스/계약을 먼저 깐다).
- 자원 (skill / agent / hook) 이 추가·삭제되어 context-map 인덱스 갱신이 필요하다.
- 문서가 길어지거나 책임이 섞여 (CLAUDE.md 가 AGENTS.md 책임 침범, README 가 작업 계약 흡수 등) 정리가 필요하다.

**When NOT to use**:

- 자원 타입 결정 (skill / agent / hook / docs 중 무엇?) → `harness-resource-design` 또는 `agent-skill-designer`.
- 개별 skill / agent / hook 파일 작성 → `skill-creator` / `agent-creator` / `hook-creator`.
- 검증 인프라 (task log / golden-set / 사이클) → `evaluation-loop-design` (예정).
- `docs/architecture.md` / `docs/domain/` 의 *본문 prose* — 본 스킬은 *skeleton* (섹션 헤더 + placeholder + 안내 코멘트) 까지만 작성. 본문 prose 는 사람 / 후속 작업.
- 외부 모델 의견 / PR 리뷰 → `codex-reviewer` / `pr-review-toolkit`.

## Workflow

### Phase 1: Inventory & Inspect

작성 *전* 에 현재 docs 트리 + 자원 상태를 점검해 중복·누락·책임 누수를 막는다.

읽는다 (있는 것만):

- `<repo>/README.md` · `<repo>/AGENTS.md` · `<repo>/CLAUDE.md` — 책임 분리 확인
- `<repo>/docs/` 전체 디렉토리 — `docs/README.md`, `docs/architecture.md`, `docs/agent/context-map.md`, `docs/agent/roles.md`, `docs/decisions/`, `docs/domain/`, `docs/integrations/`, `docs/workflows/`, `docs/security.md`
- 자원 inventory — `references/context-map-write.md` 의 §Inventory 절차 (skill / agent / hook / doc / role 스캔)

수집 항목:

| 카테고리 | 출처 | 사용 |
|---|---|---|
| 존재하는 docs | `find docs -maxdepth 2 -name "*.md"` + 루트 진입 문서 | Inventory 섹션의 *what exists* |
| 누락된 카테고리 | 위 트리에서 spec §4.2 의 9 종 중 빠진 것 | Gaps 섹션의 *missing* |
| 책임 누수 | CLAUDE.md ↔ AGENTS.md ↔ README.md 중복 / Claude 한정 문장이 AGENTS 에 / 사람용 onboarding 이 AGENTS 에 / 아키텍처 dump 가 AGENTS 에 | Gaps 섹션의 *boundary leak* |
| 자원 inventory | skill / agent / hook / role 이름 (frontmatter / settings.json) | context-map.md 작성 시 셀 인용 풀 |

각 reference 가 다루는 작성 절차:

- `references/agents-md-write.md` — AGENTS.md (도구 공통 코딩 에이전트 계약)
- `references/context-map-write.md` — `docs/agent/context-map.md` (라우팅 표)
- `references/claude-md-write.md` — `CLAUDE.md` (Claude 한정 짧은 운용 지침, Apache-2.0 attribution)
- `references/docs-tree-write.md` — `docs/architecture.md` / `docs/decisions/` / `docs/domain/` / `docs/integrations/` / `docs/workflows/` / `docs/security.md` skeleton

### Phase 2: Document Plan

Phase 1 결과를 다음 4 섹션으로 정리해 사용자에게 제시한다 (Effect gate 1단계 — CONSTITUTION §3.3).

```markdown
# Harness Installation Spec — context-map

> Generated by: context-map-architecture
> Date: <iso8601>
> Trigger: <user request>
> spec_version: v1

## Inventory
- 존재하는 docs (카테고리별)
- 자원 inventory (skill / agent / hook / role 이름)

## Gaps
- 누락된 docs 카테고리
- 책임 누수 (어느 항목이 어느 파일로 이동해야 하는지)
- ghost reference (context-map.md 가 인용하지만 inventory 에 없는 자원)

## Document Plan
- 작성 / 수정 / 이동 항목 (파일별)
- 각 파일의 골격 (어떤 섹션을 추가·변경할지)
- skeleton 만 작성하는 항목 (architecture / domain — 본문 prose 는 사람 작업)

## Applied Changes
- (Phase 3 후 채워짐 — 실제 작성/수정한 파일 목록)
```

사용자가 Document Plan 을 검토해 "진행" / "go" / "proceed" 신호를 줄 때 Phase 3 진입. "묻지 말고 진행" 이 사전 합의된 경우만 확인 없이 진행한다. Phase 2 만 실행하고 Phase 3 를 보류하는 *plan-only* 모드도 허용 (사용자가 명시 시).

### Phase 3: Apply

Document Plan 에 적힌 각 파일을 *해당 reference 의 절차를 따라* write 한다. 각 reference 는 자체 effect gate (write 직전 경로·종류·요약 1회 확인) 를 가진다 — Phase 3 는 reference 호출 wrapper.

write 순서 (의존성 기반):

1. `AGENTS.md` (있어야 다른 docs 가 인용할 *작업 계약* 이 생긴다 — `agents-md-write.md` Phase 3)
2. `CLAUDE.md` (AGENTS.md 와 책임 분리 확정 후 — `claude-md-write.md` Phase 4-5)
3. `docs/architecture.md` / `docs/decisions/` / `docs/domain/` / `docs/integrations/` / `docs/workflows/` / `docs/security.md` skeleton (`docs-tree-write.md` Phase 2)
4. `docs/agent/context-map.md` (위 모든 자원 + docs 인벤토리 확정 후 — `context-map-write.md` Phase 3)

각 write 후 즉시 verify (해당 reference 의 verify 절차) — 길이 / 인용 자원 일치 / 책임 누수 재점검.

## Output Contract

caller (사용자 또는 상위 워크플로우) 에게 반환:

```
spec: Harness Installation Spec — context-map (spec_version: v1)
mode: applied | plan-only | no-op | blocked
inventory:
  existing_docs: <count>
  resources_indexed: skills=<n>, agents=<n>, hooks=<n>, roles=<n>
gaps:
  missing_docs: <list of categories>
  boundary_leaks: <list of moves needed>
  ghost_refs: <list>
document_plan:
  create: <list of new files>
  modify: <list of edited files>
  skeleton_only: <list — architecture / domain 등>
applied_changes:
  - <file>: <action — created / edited / moved>
follow_ups:
  - <docs/architecture.md>: 본문 prose 작성 필요 (skeleton 만 들어감)
  - <docs/agent/roles.md>: 새 역할 정의 필요 (context-map 이 인용하나 정의 없음)
  - <자원이름>: context-map 이 인용하나 inventory 에 없음 — 작성 또는 표에서 제거
```

**No-op case**: 기존 docs 트리가 적절하고 자원 inventory 와 일치하면 `mode: no-op` + 변경 없음. 사용자가 명시적 갱신을 요청한 경우가 아니면 수정하지 않는다.

**Blocked case**: 자원 inventory 가 0 (skill / agent / hook 결정 자체가 안 됨) 이거나 작업 유형이 0개라 context-map 을 채울 수 없으면 `mode: blocked` + `needs_input` (예: "먼저 `resource-design` 으로 첫 자원 결정 필요").

**Plan-only case**: 사용자가 Phase 2 까지만 요청 (Document Plan 검토 후 보류) → `mode: plan-only` + spec 본문 반환. Phase 3 는 다음 호출에서.

## Common Failures

| 안티패턴 | 증상 | 수정 |
|---|---|---|
| Reference dump | 한 reference 절차를 본 SKILL.md 본문에 복사 | reference 호출만 하고 절차 본문은 references/ 에 둠 |
| Body-prose drift | `docs/architecture.md` / `docs/domain/` 의 본문 prose 까지 본 스킬이 작성 | skeleton 까지만, 본문은 follow-up 으로 보고 |
| Ghost reference | context-map.md 가 inventory 에 없는 skill / agent 를 인용 | Phase 1 inventory 와 대조, 작성 후보 또는 제거 |
| Boundary leak | CLAUDE.md 가 AGENTS.md 책임 흡수 / AGENTS.md 가 docs/architecture 흡수 | Gaps 의 boundary leak 행으로 이동 권고 |
| Phase 2 skip | Document Plan 없이 곧바로 Phase 3 write | CONSTITUTION §3.3 위반 — Phase 2 effect gate 강제 |
| Order violation | context-map 을 AGENTS.md 보다 먼저 작성 | Phase 3 의존성 순서 (1→4) 강제 |
| Plan vs Apply 혼동 | Document Plan 에 "applied" 표시 | Plan = 의도, Applied Changes = 실행 결과 |
