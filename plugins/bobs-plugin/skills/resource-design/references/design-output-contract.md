# Design Output Contract

> 본 문서는 `resource-design` skill 의 reference. 두 원본을 통합:

> - spec §4.1 (workflow doc §4 호환 형식) — Inventory / Required Resources / Resource Decisions / Execution Plan.
> - (deprecated) `agents/agent-skill-designer.md` §4 — DESIGN_SUMMARY / PROPOSED_RESOURCES / CONTRACTS / IMPLEMENTATION_ORDER / RISKS / REFERENCE_NOTES (보조 brief 형식).
>

본 reference 는 `resource-design` Phase 3 의 산출 contract 를 정의한다. 표준 spec markdown (Section 1) 이 주 출력이며, agent-skill-designer §4 의 짧은 brief 형식 (Section 2) 은 사용자가 *spec 대신 design brief 만* 원할 때 사용한다.

## Section 1 — 표준 spec 형식 (workflow doc §4 호환)

본 형식은 harness-installation-workflow.md §4 에서 정의한 공통 인터페이스를 따른다 — 다른 Phase 1 design skill (`context-map-architecture`, `evaluation-loop-design`) 와 동일한 헤더/섹션명을 사용한다.

### 1.1 공통 헤더

```yaml
spec_version: v2
generated_by: resource-design
date: <iso8601>
trigger: <user request>
mode: created | no-op | needs_input | blocked
```

`spec_version` 은 workflow doc §4 의 파싱 규칙과 동기화. v0.3 부터 *YAML fenced block* 우선 (v1 markdown-like 형식은 `${CLAUDE_PLUGIN_ROOT}/references/spec-schema.md` §6 backward compatibility 로 graceful fallback). `mode` 는 top-level discriminator — install-harness command 가 mode 별 payload 분기 (spec-schema §2 참조).

resource-design 산출 mode 는 4 종 — `created` (default, §1.2 표준 payload) / `no-op` / `needs_input` / `blocked`. `applied` 와 `plan-only` 는 resource-design 산출 mode 아님 (context-map-architecture / evaluation-loop-design 책임 — spec-schema §2.2 mapping 표). outcome-first 키워드 (CONSTITUTION §3.7.1) — design skill 의 *결과* 를 표현.

### 1.2 표준 4 섹션

```markdown
## Inventory

현재 자원 (skill / agent / hook / command / runtime settings) 인벤토리 + 사용자 작업 매핑.
- skills: <count> (path or list)
- agents: <count>
- hooks: <count>
- commands: <count>
- runtime: <settings.json scope summary>

## Required Resources

작업 유형 → 필요한 자원 결정 표.

| 작업 유형 | 필요 자원 | 신호 |
|---|---|---|
| ... | command / skill / agent / hook | ... |

## Resource Decisions

각 자원: name / type / 책임 한 줄 / 우선순위.

- **<name>** | <type> | <책임> | <P0 / P1 / P2>
  - input: <fields>
  - output: <fields>
  - failure_modes: <NEEDS_INPUT / BLOCKED / ESCALATION>
  - compatibility: <기존 호출자 영향>

## Execution Plan

target creator skill + args + rationale (Phase 2 dispatch 용 — main session 이 §4 파싱 규칙으로 처리).

1. `target: skill-creator | args: name=<x>, scope=<plugin|user|project> | rationale: <한 줄>`
2. `target: agent-creator | args: name=<x>, scope=<plugin|user|project>, subagent_type=<x?> | rationale: <한 줄>`
3. `target: hook-creator | args: name=<x>, event=<PreToolUse|...>, matcher=<x>, scope=<plugin|user|project> | rationale: <한 줄>`

workflow doc §5.1 args 표 + 3 creator §0 args 입력 contract 와 일치 (Step 7 정렬). `subagent_type` 은 선택 (main session 호출용 식별자, 누락 시 creator §0 가 결정). agent-creator 의 `model` 은 §0 intent 본질 질문 답안 (creator 가 결정) 이라 args 우회 불가 — 본 표 에서 제외.
```

### 1.3 변형 케이스 (workflow doc §4 형식)

**mode: no-op** — 기존 자원으로 충분:

```markdown
# Harness Installation Spec — resource

> Generated by: resource-design
> Date: <iso8601>
> Trigger: <user request>
> spec_version: v1
> mode: no-op

## Reasoning

<기존 자원명 + 책임 매핑 + 왜 새 자원이 불필요한지>
```

**mode: needs_input** — 사용자 추가 정보가 필요. `category` 로 사유 구분:

```markdown
> mode: needs_input
> category: design | inventory
> items:
>   - <질문 또는 누락 path>
```

호출자 분기 매핑:

| category | 사유 | 호출자 후처리 |
|---|---|---|
| `design` | 자원 타입 결정 모호 (반복 vs 일회성 / 명시 호출 vs 자동 활성화 등) | 사용자에게 의도 캡처 질문 |
| `inventory` | 접근 불가 디렉토리 / 권한 부족 / 외부 도구 정보 누락 | 사용자에게 환경 보완 요청 |

## Section 2 — 보조 brief 형식 (1,200 토큰 이내)

사용자가 *spec 출력 대신 짧은 design brief* 만 원할 때 사용한다. agent-skill-designer §4 의 풀 템플릿을 그대로 가져왔다. 모든 섹션을 채우되 해당 없는 항목은 `n/a` 로 표시한다.

```
DESIGN_SUMMARY
  objective: <1줄>
  recommendation: <create | update | merge | delete | defer>
  resource_shape: <command/skill/agent/hook/runtime_settings/plugin/mixed>
  rationale: <2-4줄>

PROPOSED_RESOURCES
- <name> | <type> | <owner responsibility>
  trigger: <description/frontmatter 핵심>
  tools/model/invocation: <해당 시>
  negative_cases: <1-3개>

CONTRACTS
- input: <필드>
- output: <필드>
- failure_modes: <NEEDS_INPUT/BLOCKED/ESCALATION 등>
- compatibility_notes: <기존 호출자 영향>

IMPLEMENTATION_ORDER
1. <낮은 위험 변경>
2. <contract/routing 변경>
3. <검증>

RISKS
- [P0/P1/P2] <위험> -> <완화책>

REFERENCE_NOTES
- <읽은 reference 파일과 적용한 규칙 ID>
```

## Execution Plan 파싱 규칙

Execution Plan 의 각 항목은 workflow doc §4 의 표준 파싱 규칙을 따른다 — main session 이 다음 형식으로 추출:

```
target=<creator skill name>
args=<key=value, ...>
rationale=<한 줄>
```

상세 파싱 규칙은 `harness-installation-workflow.md` §4 본문 참조. 본 reference 는 *spec 형식* 책임만 가진다.
