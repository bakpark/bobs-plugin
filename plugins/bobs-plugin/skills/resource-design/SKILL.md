# Resource Design

Claude harness 작업 패턴에 어떤 자원 (command / skill / agent / hook / runtime settings / plugin) 이 필요한지 결정하고, 기존 자원과의 책임 경계를 분리해 `Harness Installation Spec — resource` markdown 으로 산출하는 메타 스킬. 자원 자체는 만들지 않는다 — 후속 dispatch 는 호출자 책임.

## When to Use

- 사용자가 새 자원 작성을 요청 — *어떤 타입* 인지 결정 필요 ("새 스킬 만들어줘", "agent 필요할까", "hook 으로 자동화", "커맨드로 분리").
- 기존 자원의 책임 분리 / merge / migration 결정 ("두 스킬 합쳐줘", "이 에이전트 책임 너무 많아").
- 자원 inventory 정리 (어느 작업 유형에 어떤 자원이 매핑되는지).
- harness 전반의 자원 타입 결정 검토.

## When NOT to Use

- 자원의 *본문* 작성 · 수정 → 타입 결정 후 `skill-creator` / `agent-creator` / `hook-creator`.
- docs-tree (AGENTS.md / CLAUDE.md / docs/ index + routing 표) 설계 → `context-map-architecture`.
- 평가 인프라 (eval loop / GAP report 표준화) → `evaluation-loop-design`.
- 정적 rule 감사 (P0/P1/P2 + rule ID + confidence) → `agent-skill-auditor`.
- 코드 / PR 리뷰 → `pr-review-toolkit` / `codex-reviewer`.
- ecosystem 자동화 추천 (어떤 도구를 도입할지) → vendored `claude-automation-recommender` 단독 호출.

## Capability Procedure

본 스킬은 3 phase 로 동작한다. phase 간 산출은 main context 에 남고 마지막 phase 산출만 호출자에게 spec markdown 으로 전달한다. 세부 절차는 `references/intent-capture.md` 의 §0-§3.

### Phase 1: Inspect

자원 inventory 와 사용자 발화의 작업 컨텍스트를 수집한다.

Inspect 도메인:

- `<repo>/.claude/skills/`, `plugins/*/skills/`, `~/.claude/skills/` (skill inventory).
- `<repo>/.claude/agents/`, `plugins/*/agents/` (agent inventory).
- `<repo>/.claude/settings.json` (hook 등록 + runtime settings).
- `<repo>/.claude/commands/`, `plugins/*/commands/`, `~/.claude/commands/` (command inventory).
- 사용자 발화에서 *작업 유형* 추출 (반복 / 일회성 / 격리 필요 / 자동화 필요 / 명시 호출 / 결정론 필요 / 권한 정책).

읽을 자료:

- `references/decision-rules.md` — 5-asset taxonomy + 선택 순서 + 주제 → GUIDE 위치 index.
- `references/intent-capture.md` — 트리거 판단 + escape hatch + 입력 추출 + 결정 절차.

Phase 1 산출: inventory 요약 (어떤 자원이 이미 있고, 사용자 발화가 어디에 매핑되는지).

### Phase 2: Decision

자원 타입 결정 트리 (CONSTITUTION §4 선택 순서):

| 신호 | 결정 |
|---|---|
| 프로젝트 고유 정보 (디렉토리 구조 / 팀 convention / 특정 명령) | `CLAUDE.md` (자원 아님) |
| 권한 / 모델 / MCP / memory / budget / tool loading 정책 | runtime settings |
| 사용자가 명시 호출하는 workflow entrypoint / 인자 수집 / plan gate / 얕은 orchestration | command |
| 매 이벤트마다 결정론적 보장이 필요 | hook |
| 별도 context / 도구 권한 / 별도 model 이 필요한 specialist 역할 | agent |
| 자동 활성화되는 외부 인프라 · API · provider, 또는 도메인 능력 확장 | skill |
| 여러 자원을 배포 · 공유 단위로 묶음 | plugin |

기본값은 *기존 자원 수정* — 새 자원은 중복 호출 경로 / description 충돌 / 권한 증가가 정당화될 때만. CONSTITUTION §3.10 Overlap Must Be Intentional 과 §3.11 User-Initiated Workflows Need Commands 적용.

In-flight escape hatches — 결정 중 다른 design skill 로 전환:

| 신호 | 전환 |
|---|---|
| docs-tree (AGENTS.md / CLAUDE.md / 라우팅 표) 설계가 핵심 | `context-map-architecture` |
| 평가 loop / GAP report 표준화 | `evaluation-loop-design` |
| 정적 rule 감사만 필요 | `agent-skill-auditor` |
| ecosystem 추천만 필요 | vendored `claude-automation-recommender` |

자원 타입별 세부 패턴이 필요하면 `decision-rules.md` index 를 보고 normative GUIDE (`${CLAUDE_PLUGIN_ROOT}/references/{SKILL,AGENT,COMMAND,HOOK,RUNTIME}-GUIDE.md`) 의 해당 § 만 직접 읽는다.

Phase 2 산출: 자원별 결정 (name / type / 책임 한 줄 / 우선순위).

### Phase 3: Spec Output (Effect gate)

`Harness Installation Spec — resource` markdown 4 섹션을 생성한다. 자세한 형식 · 예제는 `references/design-output-contract.md`.

```markdown
# Harness Installation Spec — resource

> Generated by: resource-design
> Date: <iso8601>
> Trigger: <user request>
> spec_version: v1

## Inventory
## Required Resources
## Resource Decisions
## Execution Plan
```

각 섹션 책임:

- `Inventory` — Phase 1 의 자원 inventory 요약 + 사용자 작업 매핑.
- `Required Resources` — 작업 유형 → 필요한 자원 결정 표.
- `Resource Decisions` — 각 자원: name / type / 책임 한 줄 / 우선순위.
- `Execution Plan` — 후속 dispatch 대상 (`target creator skill` + `args` + `rationale`).

Effect gate (이중):

- **1단계 (design)** — spec 본문 사용자 검토 + 승인. 본 스킬은 파일 작성 부수 효과는 없지만 *Execution Plan dispatch* 라는 후속 효과의 진입점이므로 spec 자체에 gate 적용 (CONSTITUTION §3.3 Effects Require Gates).
- **2단계 (apply)** — main session 이 Execution Plan 항목별로 target creator skill (`skill-creator` / `agent-creator` / `hook-creator`) 을 순차 dispatch. 각 creator 의 §2 effect gate 가 실제 파일 write 시 다시 확인한다.

## Output Contract

호출자가 파싱 가능한 형태로 다음 중 하나를 반환한다.

**created** — spec 작성 완료:

```
mode: created
spec_path: <absolute path> | inline
spec_version: v1
resources: <N>
execution_plan_items: <N>
follow_ups: <optional — claude-automation-recommender ecosystem cite 필요 / inventory 미완 등>
```

**no-op** — 기존 자원으로 충분:

```
mode: no-op
reason: <기존 자원명 + 책임 매핑>
```

**needs_input** — 사용자 추가 정보가 필요. `category` 로 사유 구분:

```
mode: needs_input
category: design | inventory
items:
  - <질문 또는 누락 path>
```

- `category: design` — 자원 타입 결정 모호. 예: "이 작업이 *반복* 인가 *일회성* 인가?", "사용자가 직접 호출하나, 자동 활성화되나?"
- `category: inventory` — inventory 가 불완전 (접근 불가 디렉토리 / 권한 부족 / 외부 도구 정보 누락).

호출자는 항상 *사용자에게 묻는다* 라는 동일 후처리를 하되, `category` 로 질문 톤을 분기 (design = 의도 캡처, inventory = 환경 보완).

세부 spec 본문은 응답에 풀어 쓰지 말고 spec_path 만 안내한다 — main context 절약 (CONSTITUTION §3.7.1 Context Is A Managed Resource).

## Common Failures

- **Phase 1 skip** → 기존 자원과 중복되는 새 자원 권고. inventory 의 부분 누락이 원인. 4 개 도메인 (skills / agents / settings.json / commands) 모두 확인.
- **자원 타입을 *기능 친밀도* 로 결정** → command 가 적합한데 skill 로 권고. *작업 유형* (반복 / 일회성 / 격리 / 결정론 / 명시 호출) 으로 결정한다.
- **선택 순서 무시** → skill 부터 검토. CONSTITUTION §4 순서는 *CLAUDE.md → runtime → command → hook → agent → skill* — skill 은 마지막.
- **Phase 3 gate 생략** → spec 출력 후 곧바로 dispatch 권유. 호출자가 spec 검토 기회를 잃는다. 1 단계 gate 는 spec 검토, 2 단계 gate 는 creator skill 의 effect gate.
- **5 sibling design skill 과 책임 혼동** — `context-map-architecture` (docs-tree), `evaluation-loop-design` (eval 인프라), `agent-skill-auditor` (정적 감사), `claude-automation-recommender` (ecosystem 추천), `skill-creator` / `agent-creator` / `hook-creator` (자원 본문 작성). escape hatch 표 · When NOT 섹션 참조.
- **Description-as-runbook** — 본 SKILL.md 의 description 이 절차를 요약하면 호출자가 본문을 읽지 않고 description 만 따라 단축 실행. description 은 activation signal 만.
- **Rule index 본문 재생산** — references 안에 GUIDE 규칙을 복사하면 drift 발생. `decision-rules.md` 는 *index only*, 규칙 본문은 항상 normative GUIDE.

## Verification

본 스킬의 trigger / no-op / near-miss 검증 인프라는 sibling `evaluation-loop-design` 가 표준화한다 — 본 스킬은 case 목록을 본문에 두지 않는다. 회귀 의심 시 description trigger phrase 7건 (한·영 혼용) + `When NOT to Use` 6항 + escape hatch 표 4건 + Common Failures 6항이 사실상의 가벼운 검증 표면.
